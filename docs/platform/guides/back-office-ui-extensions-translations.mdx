Litium uses [ASP.NET standard localization](https://docs.microsoft.com/en-us/aspnet/core/fundamentals/localization?view=aspnetcore-6.0) with translations stored in resource files. Create custom resource files in `Src/Litium.Accelerator/Resources` (create the folder if needed).

> When multiple resource files contain the same key, the last one in the build order wins.

Resource files follow the naming pattern `Name[.culture].resx`. `Name` can be any identifier related to your addâ€‘on. `culture` is optional and represents the language (for example, `sv-SE`).

- `Common.resx` usually stores fallback (often English) translations.
- `Common.sv-se.resx` stores Swedish translations. If a translation is missing, the fallback file is used.

Include the resources in `Litium.Accelerator.csproj`:

```xml
<ItemGroup>
  <LitiumLocalization Include="Resources\*.resx" />
</ItemGroup>
```

Add the `Litium.Localization.UseAdministrationResource` assembly attribute to make the strings available via `IStringLocalizer`:

```xml
<ItemGroup>
  <AssemblyAttribute Include="Litium.Localization.UseAdministrationResource" />
</ItemGroup>
```

The Accelerator ships with a single `.resx` file containing strings for all areas. Using resource files in add-ons simplifies installation because no XML needs to be imported in backoffice. Use unique keys, for example `SpecialWidgetName`, to avoid collisions.

## Usage

You can retrieve translations in the backoffice in two ways:

1. Use `AsAngularResourceString()` from `Litium.Web.LocalizationExtensions`, e.g. `"your.resource.key".AsAngularResourceString()`.
2. Inject `IStringLocalizer` into a class and call `_localizer.GetString("your.resource.key")`.

Angular components rely on `ngx-translate`. Refer to the [ngx-translate documentation](https://github.com/ngx-translate/core) for usage details.
