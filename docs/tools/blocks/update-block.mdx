## Overview

Update an existing content block in your Litium instance. This tool allows you to modify block content, fields, and other properties.

<Warning>
  This tool **modifies data** in Litium. Always verify the system ID and changes before updating.
</Warning>

## Parameters

<ParamField path="systemId" type="string" required>
  The unique system ID of the block to update
  
  **Example**: `"abc-123-def-456"`
</ParamField>

<ParamField path="block" type="object" required>
  Updated block data (only include fields you want to change)
  
  <Expandable title="Block Object Properties">
    <ParamField path="name" type="string" optional>
      New display name for the block
    </ParamField>
    
    <ParamField path="content" type="object" optional>
      Updated block content (HTML, text, etc.)
    </ParamField>
    
    <ParamField path="fields" type="object" optional>
      Updated custom field values
    </ParamField>
  </Expandable>
</ParamField>

## Response

Returns the updated block with all modifications applied.

<ResponseField name="systemId" type="string">
  System ID of the updated block
</ResponseField>

<ResponseField name="name" type="string">
  Updated name of the block
</ResponseField>

<ResponseField name="content" type="object">
  Updated content data
</ResponseField>

<ResponseField name="modified" type="string">
  Modification timestamp (ISO 8601 format)
</ResponseField>

<ResponseField name="modifiedBy" type="string">
  Username of the person who made the update
</ResponseField>

## Common Workflows

<Steps>
  <Step title="Find the Block">
    Use [search_blocks](/tools/blocks/search-blocks) or [get_block](/tools/blocks/get-block) to find the block to update
  </Step>
  <Step title="Review Current Content">
    Use [get_block](/tools/blocks/get-block) to see the current block content
  </Step>
  <Step title="Prepare Updates">
    Prepare the updated content or field values
  </Step>
  <Step title="Update Block">
    Use `update_block` with the systemId and new data
  </Step>
  <Step title="Verify Changes">
    Use [get_block](/tools/blocks/get-block) again to confirm the update was successful
  </Step>
</Steps>

## Partial Updates

<Note>
  You only need to include the fields you want to change. Unchanged fields will retain their current values.
</Note>

Example of updating only the background color:
```json
{
  "systemId": "block-123",
  "block": {
    "fields": {
      "backgroundColor": "#00ff00"
    }
  }
}
```

The `name` and `content` fields remain unchanged.

## Troubleshooting

<AccordionGroup>
  <Accordion title="Block not found">
    **Cause**: The system ID doesn't exist
    
    **Solution**: Verify the systemId using [search_blocks](/tools/blocks/search-blocks) or [get_block](/tools/blocks/get-block)
  </Accordion>

  <Accordion title="Validation error">
    **Cause**: Updated data doesn't meet validation requirements
    
    **Solution**: Ensure HTML is well-formed and all required fields are present
  </Accordion>

  <Accordion title="No changes applied">
    **Cause**: Update data is empty or identical to current values
    
    **Solution**: Ensure you're sending actual changes in the `block` object
  </Accordion>

  <Accordion title="Permission denied">
    **Cause**: Insufficient permissions to update blocks
    
    **Solution**: Verify your OAuth2 credentials have write permissions
  </Accordion>

  <Accordion title="Concurrent modification error">
    **Cause**: Block was modified by another user during your update
    
    **Solution**: Refresh the block data with [get_block](/tools/blocks/get-block) and try again
  </Accordion>
</AccordionGroup>

## Best Practices

<Check>
  **Get before update**: Always retrieve current block data before updating to avoid overwriting changes
</Check>

<Check>
  **Partial updates**: Only send fields that need to change to preserve other content
</Check>

<Check>
  **Verify changes**: Use [get_block](/tools/blocks/get-block) after updating to confirm success
</Check>

<Check>
  **Test updates**: Test changes in a staging environment before production
</Check>

<Check>
  **Backup content**: Keep a copy of original content before making significant changes
</Check>
